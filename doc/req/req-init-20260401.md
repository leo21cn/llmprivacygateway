## 1. 产品背景与目标

### 1.1 产品背景

#### 1.1.1 市场痛点分析

随着 AI 编程、AI 龙虾等桌面应用的普及，用户在使用这些应用时会将大量数据发送到 LLM 服务提供商。这带来了严重的隐私风险：

- **敏感数据泄露风险**：代码中的 API Key、数据库密码、用户个人信息等可能被意外传输到云端
- **合规性挑战**：企业需要遵守 GDPR、数据本地化等法规，对数据处理有严格要求
- **现有方案局限**：云端代理方案存在信任问题，网络延迟影响体验

#### 1.1.2 目标用户画像

| 用户类型 | 描述 | 核心需求 |
|----------|------|----------|
| 企业开发者 | 在企业环境中使用 AI 编程工具 | 保护代码资产、满足合规要求 |
| 个人开发者 | 使用 AI 辅助编程的自由职业者 | 保护个人项目隐私、API Key 安全 |
| 安全团队 | 负责企业数据安全的专业人员 | 审计追踪、安全策略执行 |
| 合规专员 | 负责企业合规的管理人员 | 满足法规要求、生成合规报告 |

#### 1.1.3 竞品分析

| 方案类型 | 优点 | 缺点 |
|----------|------|------|
| 云端 API 网关 | 部署简单、功能丰富 | 数据需传输到第三方、存在信任风险 |
| 网络层代理 | 透明接入 | 配置复杂、缺乏智能识别 |
| LLM 内置安全 | 原生支持 | 覆盖有限、不可自定义 |

#### 1.1.4 核心技术参考：微软 Presidio

本项目将基于微软开源的 **Presidio** 项目构建核心 PII 识别和脱敏能力。

**Presidio 项目概述：**
- **项目定位**：上下文感知、可插拔、可定制的 PII 脱敏服务
- **开源协议**：MIT License
- **GitHub**：https://github.com/microsoft/presidio

**Presidio 核心能力：**

| 能力 | 说明 |
|------|------|
| PII 识别 | 支持 NER、正则表达式、规则逻辑、校验和验证等多种识别技术 |
| PII 脱敏 | 支持替换、掩码、加密、伪名化、Faker 生成等多种脱敏策略 |
| 多语言支持 | 支持多语言 PII 识别 |
| 图像处理 | 支持标准图像和 DICOM 医学图像中的文本脱敏 |
| 可扩展 | 支持自定义识别器和脱敏操作符 |
| REST API | 提供完整的 HTTP API 接口 |

**Presidio 主要组件：**

| 组件 | 功能 |
|------|------|
| presidio-analyzer | PII 分析识别模块（核心识别引擎） |
| presidio-anonymizer | PII 匿名化处理模块（脱敏执行引擎） |
| presidio-image-redactor | 图像脱敏模块 |
| presidio-structured | 结构化数据处理模块 |

**整合方案：** 采用嵌入式后台服务模式，将 Presidio 作为本地 HTTP 服务打包到 macOS 应用中，随应用一起启动和管理。

#### 1.2.1 核心价值主张

**"本地化的智能隐私卫士"**

- **透明代理体验**：通过虚拟 API Key 机制，无需修改现有应用配置
- **智能识别能力**：结合正则、关键词、AI 多种检测方式，平衡准确性与灵活性

## 2. 功能需求

### 2.1 核心代理模块

#### 2.1.1 API 请求拦截与转发

**功能描述：**
应用作为本地代理服务器，拦截通过虚拟 API Key 发起的请求，进行敏感信息处理后转发到真实的 LLM API 端点。

**功能要点：**

| 功能项 | 详细说明 |
|--------|----------|
| 支持的 API 协议 | OpenAI API 格式（兼容大多数 LLM 服务） |
| 支持的端点 | /v1/chat/completions, /v1/completions, /v1/embeddings 等 |
| 流式响应 | 支持 SSE（Server-Sent Events）流式处理 |
| 超时机制 | 可配置的请求超时时间，默认 60 秒 |
| 重试策略 | 支持指数退避重试，可配置重试次数 |

**边界条件：**
- 网络异常时返回友好错误信息，记录日志
- API 服务不可用时支持降级策略
- 最大并发连接数限制（默认 100）

#### 2.1.2 虚拟 API Key 管理

**功能描述：**
生成和管理虚拟 API Key，建立虚拟 Key 与真实 Key 的映射关系。

**功能要点：**

| 功能项 | 详细说明 |
|--------|----------|
| Key 生成 | 生成格式为 `sk-virtual-xxxxxx` 的虚拟 Key |
| Key 映射 | 虚拟 Key 与真实 Key 一对一映射 |
| 权限控制 | 可配置虚拟 Key 的权限范围（模型、端点限制） |
| 生命周期 | 支持设置过期时间，支持手动吊销 |
| 使用统计 | 记录每个虚拟 Key 的使用次数和时间 |

#### 2.1.3 请求/响应处理管道

**功能描述：**
定义清晰的处理管道，对请求和响应内容进行有序处理。

**处理流程：**

```
请求流程：
接收请求 → 验证虚拟 Key → 替换真实 Key → 敏感信息检测 → 脱敏处理 → 转发请求

响应流程：
接收响应 → 敏感信息还原（可选） → 返回给客户端 → 记录审计日志
```

#### 2.2.3 自定义识别器

**功能描述：**
支持通过 Presidio 的自定义识别器机制扩展检测能力。

**识别器类型：**

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| Deny-list recognizers | 拒绝列表识别器 | 关键词黑名单匹配 |
| Regex recognizers | 正则识别器 | 模式匹配（项目代号、内部标识等） |
| Rule-based recognizers | 基于规则的识别器 | 复杂逻辑判断 |
| External services | 外部服务识别器 | 调用第三方检测服务 |
| Ad-hoc recognizers | 临时识别器 | 一次性检测需求 |

**自定义识别器配置示例：**

```json
{
  "name": "ProjectCodeRecognizer",
  "supported_entity": "PROJECT_CODE",
  "patterns": [
    {
      "name": "PRJ_pattern",
      "regex": "PRJ-\\d{3}",
      "score": 0.9
    }
  ],
  "context": ["项目", "代号", "编号"],
  "supported_language": "zh"
}
```

#### 2.2.4 识别器配置与优先级

**配置管理：**
- 可通过 YAML 文件配置识别器注册表（No-code 配置）
- 支持运行时动态加载自定义识别器
- 识别器结果置信度阈值可配置

**优先级与评分：**
- 每个识别器输出 `score`（0-1）表示置信度
- 可配置最低置信度阈值（默认 0.5）
- 多识别器结果合并策略可配置

### 2.3 脱敏处理模块（基于 Presidio Anonymizer）

**架构说明：**
脱敏处理模块基于微软 Presidio Anonymizer 构建，通过本地 HTTP 服务调用实现 PII 脱敏。

#### 2.3.1 Presidio Anonymizer 集成

**服务调用方式：**

```
macOS 应用                              Presidio 服务
    │                                       │
    │  POST /anonymize                      │
    │  {                                    │
    │    "text": "原始文本",                 │
    │    "anonymizers": {                   │
    │      "EMAIL_ADDRESS": {               │
    │        "type": "mask",                │
    │        "masking_char": "*",           │
    │        "chars_to_mask": 4             │
    │      }                                │
    │    }                                  │
    │  }                                    │
    │ ─────────────────────────────────────▶│
    │                                       │ Anonymizer 处理
    │                                       │
    │  {                                    │
    │    "text": "脱敏后文本",               │
    │    "items": [                         │
    │      {"operator": "mask",             │
    │       "entity_type": "EMAIL_ADDRESS", │
    │       "start": 10, "end": 30,         │
    │       "text": "****@email.com"}       │
    │    ]                                  │
    │  }                                    │
    │ ◀─────────────────────────────────────│
```

#### 2.3.2 脱敏操作类型（Anonymizer Operators）

| 操作类型 | 描述 | 配置示例 | 适用场景 |
|----------|------|----------|----------|
| **replace** | 替换为固定文本 | `"type": "replace", "new_value": "<EMAIL>"` | 通用脱敏 |
| **mask** | 字符掩码 | `"type": "mask", "masking_char": "*", "chars_to_mask": 4, "from_end": true` | 部分隐藏 |
| **redact** | 完全移除 | `"type": "redact"` | 敏感内容删除 |
| **hash** | 哈希替换 | `"type": "hash", "hash_type": "sha256"` | 可追溯脱敏 |
| **encrypt** | 加密存储 | `"type": "encrypt", "key": "encryption_key"` | 可还原脱敏 |
| **decrypt** | 解密还原 | `"type": "decrypt", "key": "encryption_key"` | 响应还原 |
| **custom** | 自定义函数 | `"type": "custom", "lambda": lambda_func` | 特殊处理 |

#### 2.3.3 脱敏策略配置

**默认脱敏策略映射：**

| PII 类型 | 默认策略 | 脱敏效果示例 |
|----------|----------|--------------|
| EMAIL_ADDRESS | mask | `user@email.com` → `****@email.com` |
| PHONE_NUMBER | replace | `13812345678` → `<PHONE>` |
| CREDIT_CARD | mask | `4111111111111111` → `4111****1111` |
| PERSON | replace | `张三` → `<PERSON>` |
| CN_ID_CARD | replace | `身份证号` → `<ID_CARD>` |
| IP_ADDRESS | replace | `192.168.1.1` → `<IP>` |
| URL | mask | `https://secret.com` → `https://*****.com` |
| CRYPTO | replace | `钱包地址` → `<CRYPTO>` |

**上下文感知策略：**

| 内容类型 | 特殊策略 | 说明 |
|----------|----------|------|
| 代码中的 API Key | `replace: "YOUR_API_KEY"` | 保持代码可执行性 |
| JSON 配置 | `replace: "***"` | 保持 JSON 结构完整 |
| SQL 语句 | `replace: "?"` | 保持 SQL 语法有效 |

#### 2.3.4 脱敏还原机制

**功能描述：**
使用 Presidio 的 encrypt/decrypt 操作实现可逆脱敏，支持在 LLM 响应中还原原始信息。

**实现流程：**

```
请求阶段：
1. 检测到敏感信息（如 user@email.com）
2. 使用 encrypt 操作加密原始值
3. 记录加密密钥和映射关系（内存存储）
4. 发送脱敏后的请求到 LLM

响应阶段：
1. 接收 LLM 响应
2. 查找脱敏映射表
3. 使用 decrypt 操作还原原始值
4. 返回还原后的响应给应用
5. 清除本次请求的映射记录
```

**安全考量：**
- 加密密钥仅在请求生命周期内有效
- 映射表存储在安全内存区域
- 请求结束后立即清除映射数据
- 支持配置是否启用还原功能

#### 2.3.5 自定义脱敏操作符

**功能描述：**
支持开发自定义脱敏操作符，满足特殊业务需求。

**实现方式：**
- 通过 Python Lambda 函数定义自定义逻辑
- 支持调用 Faker 库生成假数据
- 可集成外部服务进行脱敏处理

#### 2.5.3 规则更新机制

**更新策略：**

| 更新类型 | 触发条件 | 更新方式 | 通知方式 |

### 4.2 Presidio 嵌入式服务设计

#### 4.2.1 服务打包与部署

**打包方式：**

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **Docker 嵌入** | 将轻量级 Docker 镜像打包到应用中 | 环境隔离、部署简单 | 需用户安装 Docker |
| **Python 嵌入** | 将 Python 环境和 Presidio 打包到应用 | 无外部依赖 | 包体积较大 |
| **混合方案** | 首次启动时自动安装配置 Presidio 服务 | 灵活、可控 | 需要网络下载 |

**推荐方案：Python 嵌入 + 自动安装**

```
应用启动流程：
1. macOS 应用启动
2. 检查内置 Python 环境
3. 启动 Presidio 服务进程
4. 等待服务就绪（健康检查）
5. 开始接收代理请求
```

#### 4.2.2 服务生命周期管理

**服务状态管理：**

| 状态 | 说明 | 用户界面指示 |
|------|------|--------------|

## 7. 安全性设计

### 7.1 威胁模型

| 威胁类型 | 描述 | 风险等级 | 防护措施 |
|----------|------|----------|----------|
| API Key 泄露 | 真实 API Key 被恶意程序读取 | 高 | Keychain 存储、沙盒隔离 |
| 内存攻击 | 敏感数据在内存中被读取 | 高 | 最小化驻留、安全擦除 |
| 日志泄露 | 审计日志包含敏感信息 | 中 | 日志脱敏、加密存储 |
| 中间人攻击 | 本地代理被劫持 | 中 | 本地绑定、证书固定 |
| 规则篡改 | 恶意修改检测规则 | 中 | 规则加密、完整性校验 |

### 7.2 加密方案

#### 7.2.1 密钥层次

```
主密钥 (MK) [存储于 Keychain]
    │
    │ 派生 (HKDF)
    ▼
数据加密密钥 (DEK) [用于加密 SQLite 数据]
    │
    │ 派生
    ▼
日志加密密钥 (LEK) [用于加密日志文件]

会话密钥 (SK) [每次启动生成，用于内存加密]
```

#### 7.2.2 加密算法

| 场景 | 算法 | 说明 |
|------|------|------|
| 数据加密 | AES-256-GCM | 认证加密，防篡改 |
| 密钥派生 | HKDF-SHA256 | 安全的密钥派生 |
| 密码哈希 | Argon2id | 抗暴力破解 |
| TLS | TLS 1.3 | 通信安全 |
| 数据完整性 | HMAC-SHA256 | 完整性校验 |

### 7.3 安全存储流程

**API Key 存储流程：**
1. 用户输入真实 API Key
2. 生成随机盐值
3. 使用 Keychain 存储主密钥
4. 使用主密钥派生数据加密密钥
5. 使用 DEK 加密 API Key
6. 存储加密后的 API Key 和盐值

**API Key 读取流程：**
1. 从 Keychain 获取主密钥
2. 派生数据加密密钥
3. 解密 API Key
4. 使用后立即擦除内存

### Context

- 技术验证：验证核心架构和 Presidio 集成方案
